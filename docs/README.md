# OctoAcme Project Management Docs

## Overview

OctoAcme follows a **structured, lifecycle-based approach to project management** grounded in customer-first principles and iterative delivery. The framework encompasses five key phases:

1. **Initiation** — Validate business need and align stakeholders around a lightweight Project One-pager
2. **Planning** — Break work into shippable increments with clear acceptance criteria and estimates
3. **Execution** — Coordinate delivery through defined team rhythms, workflows, and quality assurance
4. **Release** — Deploy with confidence through standardized pre-release requirements and verification
5. **Close & Retrospective** — Capture learnings and feed improvements back into processes

### Key Characteristics

- **Clear Ownership**: Designated Project Manager (PM) and Product Manager (PdM) roles ensure accountability
- **Defined Team Rhythm**: Daily standups, weekly syncs, and twice-weekly team meetings maintain visibility and alignment
- **Quality-First Execution**: Embedded testing, security scanning, and manual QA validate readiness before release
- **Structured Risk Management**: Risk Register tracking with escalation paths (Level 1 team → Level 2 lead → Level 3 sponsor)
- **Continuous Improvement**: Regular retrospectives capture what went well and actionable improvements
- **Transparent Communication**: Weekly stakeholder updates and single source of truth reduce ambiguity

## Core Team Roles

| Role | Responsibilities | Key Goals |
|------|------------------|-----------|
| **Project Manager (PM)** | Coordinates delivery, schedules, risks, and communications | Deliver on time, within scope, with transparency |
| **Product Manager (PdM)** | Defines outcomes, prioritizes backlog, measures success | Maximize customer value and impact |
| **Developers** | Implement features, collaborate on design and testability | Deliver reliable, maintainable code efficiently |
| **QA/Testing** | Validate quality and acceptance criteria | Ensure features meet standards before release |

## Quick Start

1. **Starting a new project?** Begin with [Project Initiation](./octoacme-project-initiation.md) to define the problem, align stakeholders, and create a One-pager.
2. **Ready to build?** Move to [Project Planning](./octoacme-project-planning.md) to break work into shippable increments and set up your team rhythm.
3. **Delivering features?** Reference [Execution & Tracking](./octoacme-execution-and-tracking.md) for daily standups, workflows, and quality gates.
4. **Going to production?** Use [Release & Deployment](./octoacme-release-and-deployment.md) to ensure readiness and reduce risk.
5. **Wrapping up?** Schedule a [Retrospective](./octoacme-retrospective-and-continuous-improvement.md) to capture learnings and improve next time.

## Key Principles

- **Customer-first**: Prioritize customer value and usability in all decisions
- **Iterative delivery**: Deliver small, testable increments regularly
- **Clear ownership**: Each project has named PM and Product Lead with explicit responsibilities
- **Data-informed decisions**: Measure impact and iterate based on evidence
- **Psychological safety**: Encourage feedback, learning, and continuous improvement

## How to Use These Docs

- **Keep your Project Charter updated** in your project repository (`README.md` or `.copilot/`)
- **Reference process-specific docs** when starting a new phase (initiation, planning, execution, etc.)
- **Add artifacts to your project** (One-pager, Risk Register, meeting notes) to maintain a single source of truth
- **Standardize communication** using templates provided in each process document
- **Escalate risks and blockers** following the structured paths defined in each phase
